doGetBean() 方法分析

看完源码，咱们再来总结下调用 getBean() 方法后，都经历了些什么：

1 getBean() 啥也不做，直接调用 doGetBean()方法;
2 转换 name ，获取真正的 beanName;
3 从缓存中获取 bean 实例
4 若获取到了，则调用 getObjectForBeanInstance() ，根据 name 返回相应的 bean 实例，即返回 FactoryBean 或 普通 bean
5 若没获取到，则递归的去父容器中查找 beanName 对有的 bean 实例， 若父容器存在，则直接返回
6 若父容器也不存在，则真的需要新建 bean 实例了，先合并 BeanDefinition
7 关联depends-on所有依赖
8 根据作用域创建 bean 实例 ，并加入缓存
9 再调用 getObjectForBeanInstance() ，根据 name 返回相应的 bean 实例 ，与步骤四作用相同
10 按需转换 bean 类型，并返回转换后的 bean 实例

1.处理name中的 ‘&’;

getSingleton(beanName) 方法分析

功能：试图从缓存或实例工厂中获取Bean

getSingleton(beanName) 方法主要是从缓存中获取 bean 实例，步骤如下：

1.从 singletonObjects 中获取bean实例
2.从 earlySingletonObjects 中获取bean实例
3.从 singletonFactories 中获取 ObjectFactory,再调用ObjectFactory.getObject()来创建bean实例,并放到earlySingletonObjects中去，并且从singletonFactories中移除掉该ObjectFactory。

这里介绍下源码中涉及到了几个缓存集合，作用如下：

缓存集合 用途
singletonObjects 用于存放完全初始化好的 bean,从该缓存中取出的 bean 可以直接使用
earlySingletonObjects 用于存放还在初始化中的 bean,用于解决循环依赖
singletonFactories 用于存放 bean 工厂,bean 工厂所产生的 bean 是还未完成初始化的 bean
registeredSingletons 用于存放当前所有已经注册的bean

这几个集合在后面 addSingleton() 也会看到，一个是往里面添加，一个是从里面获取。

getObjectForBeanInstance() 方法分析

功能：从 FactoryBean 中获取 bean 实例。
先举个栗子，看什么时候才会从 FactoryBean 中获取实例：

FactoryBean 是 Spring 容器提供的一种可以扩展容器对象实例化逻辑的接口。跟 BeanFactory 有本质性的区别， 它就是个 Bean ，是一个能生成对象或者修饰对象的工厂 Bean。当某些对象的实例话过程过于烦琐，通过XML配置过于复杂，或者某些第三方库不能直接注册到 Spring 容器中的时候，我们就可以选择实现 FactoryBean 接口来实例化这些对象。

getObjectForBeanInstance() 方法从 FactoryBean 中获取 bean 的步骤如下：

1.判断 beanInstance 的类型，如果是普通 bean，则直接返回
2.如果 bean 是 FactoryBean 类型，则先从缓存里获取 FactoryBean 生成的实例 ，若缓存中存在则直接返回
3.若缓存中不存在，则先判断 FactoryBean 是否为单例模式
4.如果 FactoryBean 为单例，则调用 FactoryBean.getObject() 方法生成实例，并放入缓存中
5.如果 FactoryBean 不是单例，则调用 FactoryBean.getObject() 方法生成实例，不需加入缓存
6.进行后置处理
7.返回 bean 实例

getMergedLocalBeanDefinition() 方法分析

功能：根据 Bean 的名字取得 BeanDefinition 并合并父子 BeanDefinition
上面源码中我已经举了例子，这个方法是用来支持标签中使用parent属性配置父类 bean ，这样字类 bean 可以继承父类 bean 的配置信息。这里我们就直接分析源码。

getMergedLocalBeanDefinition() 方法合并父子 BeanDefinition 步骤：

1 检查缓存是否存在已合并的 BeanDefinition ，有则直接返回
2 若不存在，则锁定缓存，获取 bean 标签中 parent 属性
3 若 parent 属性为空，则直接把当前 BeanDefinition 设置为 RootBeanDefinition
4 若不为空，则递归合并父类和爷爷辈 的 BeanDefinition ，直到标签中配置的 parent 的 beanName 与子类 beanName 相同，最后返回父容器的 BeanDefinition
5 以父类 BeanDefinition 的配置信息为蓝本创建 RootBeanDefinition，并填充子类配置的属性，若与父类重复则覆盖
6 设置 scope 属性，默认为 singleton
7 将合并后的 RootBeanDefinition 加入缓存，并返回

getSingleton(beanName ,ObjectFactory) 方法分析

功能：通过匿名工厂类对象创建对象

这个方法里面不仅只是简单的调用 createBean() 方法创建 bean 实例，还需要在缓存中记录 bean 的状态以及把生成的 bean 实例添加到缓存中。

getSingleton(beanName ,ObjectFactory) 方法生成 bean 实例的步骤如下：

1 先从缓存中获取单例 bean ，若存在则直接返回
2 若缓存中不存在 ，则记录 beanName 正在加载状态
3 通过 getObject() 方法调用 createBean() 创建bean
4 移除第二步骤记录的 beanName 正在加载状态
5 将新建的 bean 实例加入缓存 ，并返回实例

总结

这一章节主要讨论了 IoC 容器获取 bean 实例的内容。主要涉及 beanName 处理 、从缓存中获取 bean 实例 、通过 FactoryBean 生成 bean 实例、 合并 BeanDefinition 、解决依赖 、创建实例等方面，由于初次读源码，还有很多没有分析到位，后续再慢慢补充吧。由于getBean()方法包含的内容太多，而且它包含的 createBean() 也很复杂，我们就放在下一章节介绍 。

源码分析

配置文件的封装

Spring 的配置文件读取是通过 ClassPathResource 进行封装的。其实在 Spring 中对不同来源的文件都有对应的实现。
比如 文件资源 - FileSystemResource ; ClassPath资源 - ClassPathResource ; URL资源 - UrlResource;InputStream资源 - InputStreamResource; Byte数组 - ByteArrayResource 等等。 Spring 对这些内部资源的实现了自己的抽象结构： Resource 接口来封装底层资源。

在日常开发中，我们也可以直接使用 Spring 提供的资源文件接口。
当通过 Resource 相关类完成了对配置文件的封装后，资源加载就交给 XmlBeanDefinitionReader 处理了。

资源加载

资源的加载是在 XmlBeanFactory 初始化的时候进行的，XmlBeanFactory 的初始化很多种办法，咱们只分析使用 Resource 实例作为构造参数的办法。

XmlBeanFactor 初始化过程：

1 使用 ClassPathResource 对资源文件进行分装
2 XmlBeanFactory 初始化时，把资源 XmlBeanDefinitionReader 处理
3 设置资源的编码，对资源进行封装，作用是设置输入流编码时直接获取资源编码
4 将 XML 资源封装成 InputSource，专门用于解析 XML
5 将资源文件解析成 Document 对象
6 解析并注册Bean

总结

这一章节主要讨论了配置文件的封装以及对资源的解析。主要涉及 Resource 接口 、资源文件解析成 Document 对象 、注册 BeanDefinitions 等方面的内容。这里面主要就是对XML文件的解析，我就先不对源码进行分析了。在注册 BeanDefinitions 时，Spring 把逻辑处理交给了 BeanDefinitionDocumentReader 类处理，这里很好的应用了面向对象中单一职责的原则。

Java中Long值判断相等问题

发表于 2018-12-29 | 更新于 2019-05-28 | 分类于 问题解决

背景

公司有两套子系统，两个系统里面都有合同对象，今天需要改个子系统的判断条件：比较合同A的serial_no是否等于合同B的serial_no，于是就直接使用a.getSerial_no() == b.getSerial_no()，但是神奇的发现两个系统的返回值是不同。后来才发现坑爹的两个系统的合同类(Bean)的编号属性(serial_no)一个使用的是Integer类型，另一个使用的是Long类型，Long类型的比较返回false。为什么呢？一脸懵逼 。。。 下面我们就来了解下Long类型判断是否相等。

实验

我们先来看两段代码：

程序输出：ture

程序返回：false
What？？？ 为什么同样是Long类型，同样是 == ，结果还不同呢？
根据网上查询的结果，去查看了下源码方才知晓：

Long 中有一个LongCache内部类，在类初始化的时候，会生成一个final的static的Long类型数组，数组范围是[-128 ,127]。

Long类型的valueOf方法：当调用valueOf方法时，Java先判断欲生成的Long对象是否存在于LongCache.cache数组范围内，如果是，则直接返回存在于cache数组中的Long对象引用。否则重新新建一个Long对象。所以我们也就不难知道，当Long类型的数值大于127或小于-128时，判断两个封装类==时，会返回false。

同时Long重写了equals方法：

先通过.longValue()方法获取Long对象的基本类型long的值之后再做比较。

纠结了下。。。
LongCache是拿来干嘛的呢？为啥Integer没有呢？
LongCache的引入来节省内存和提高性能。长整型对象通过使用相同的对象引用实现了缓存和重用。
打开Integer的源码，不看不知道，一看吓一跳，Integer也有IntegerCache，只是我另外一个系统数据没有找过cache数组的范围，吓得我赶紧回去改了判断条件。

网上搜索了下，实际上Byte、Integer、Short、Long、Character都具有这种缓存机制。但是Integer可以指定缓存范围，Integer的缓存上界high可以通过jvm参数-XX:AutoBoxCacheMax=size指定，取指定值与127的最大值并且不超过Integer表示范围，而下界low不能指定，只能为-128。

总结

  • 只能用“==”只能用于比较-128至127之间的值
  • Long重写了equals方法，先通过.longValue()方法获取Long对象的基本类型long的值之后再做比较。
  • 综上所述：推荐使用equals进行比较
  • 不仅存在LongCache，Byte，Short，Character，Integer都有类似的缓存机制。

Ajax请求响应状态status为canceled

发表于 2018-11-07 | 更新于 2019-05-28 | 分类于 问题解决

需求：业务数据提交成功之后，根据表单‘项目阶段’字段的值发送邮件；
我的实现逻辑是这样的：在业务数据提交成功后的回掉函数中发起发送邮件的请求，然后关闭表单页面。

$.ajax({
    url: url,
    type: 'post',
    data: {...},
    dataType: 'json',
    success: function (result) {
        $.ajax({
            url: url_email,
            type: 'post',
            data: {...},
            dataType: 'json',
            success: function () {},
            error: function () {}
        });
        refreshGrid();//关闭当前页面，且刷新上一页面
    },
    error: function () {}
});

我发现发送邮件的Ajax请求执行后返回了一个canceled(状态码)，无法成功请求后台；

网上找了很多资料，解释有如下几种情况：
1、由于form action事件与绑定与submit按钮上的click事件同时触发，form action将表单内容以参数的形式追加到了url的末尾,而url变更则会导致页面重新加载,这就导致post请求在执行后就立即被终止！！结论：在使用到ajax的时候，尽量不要在form内使用submit进行提交。
2、场景未知，结论：ajax请求默认是异步的， 将请求改为同步即可(async: false)

我将ajax请求改成同步请求之后，确实能解决该问题，但是我发送邮件的请求业务是额外业务，如果发送邮件的时间过长，会影响我业务表单提交的速率。而且我不清楚为何异步请求会导致我的状态码变成canceled。不甘心。。。

又经过一顿操作，我发现如果不关闭表单页面，即使发送邮件的请求是异步的也没关系，这时才恍然大悟：
由于发送邮件的请求是异步的，而且紧接着又关闭了表单页面，可能还没等我发送邮件请求的请求完，页面就关闭了，此时浏览器会默认该请求也不需要，所以才取消该请求。如果将发送邮件改成同步，关闭页面的操作得等请求结束后才执行，所以就不会有问题。

现在明白了我没办法在前台使用异步的ajax请求，只能乖乖在后台新建一个线程来发送邮件实现异步的操作:

public String pushEmail(HttpServletRequest request)  {
    ...//数据处理
    new Thread(new Runnable() {
            @Override
            public void run() {
                try {
                    mailUtil.sendMailToMultiReceiver(addrs, title, text);
                } catch (MessagingException e) {
                    System.out.println("*****************邮件发送报错****************");
                    e.printStackTrace();
                }
            }
        }).start();
}

简单介绍Vuex

发表于 2018-10-23 | 更新于 2018-11-26 | 分类于 前端学习

每一个Vuex应用核心就是store（仓库）：它包含着应用中大部分的状态(state)。Vuex和单纯的全局对象有以下两点不同：
1 Vuex的状态存储是响应式的。当Vue组件从store中读取状态的时候，若store中的状态发生了变化，那相应的组件也会相应地得到高效更新
2 不能直接改变store中的状态。改变store中的状态的唯一途径就是显示的提交(commit) mutation

其实，简单的理解在 vuex 就是把需要共享的变量全部存储在一个对象里面，然后将这个对象放在顶层组件中供其他组件使用。

vuex包含有五个基本的对象:

  • state: 存储状态；可理解为类的属性
  • getters: 派生状态；可理解为类中get方法；有两个可选参数：state、getters分别可以获取state中的变量和其他的getters
  • mutations：提交状态修改；可理解为类中set方法；这是vuex中唯一修改state的方式，但不支持异步操作。第一个参数默认是state
  • actions：类似于mutations，不过actions支持异步操作。action提交的是mutation，而不是直接变更状态。第一个参数默认是和store具有相同方法和属性的context 对象，因此你可以调用 context.commit 提交一个 mutation，或者通过 context.state 和 context.getters 来获取 state 和 getters。
  • modules：store的子模块，子模块也拥有自己的 state、mutation、action、getter、甚至是嵌套子模块
